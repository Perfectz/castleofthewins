# Progression Series

## Step 00: Baseline Progression Atlas

- Folder path: `artifacts/progression-series/step-00-baseline-progression-atlas/`
- Three-sentence summary: This step establishes the current baseline for progression using code inspection plus live Playwright captures rather than redesign assumptions. It documents how race, class, level-ups, spells, perks, relics, boons, town services, mastery, contracts, commendations, and town unlocks currently interact across a run and across multiple runs. The after-side artifacts reorganize that baseline into annotated atlas boards, a system map, and build-path summaries so later redesign work has a fixed, publication-ready reference point.
- Key progression systems documented: character creation, stat formulas and growth, spell acquisition and management, perk choice, relic choice, boon choice, objective reward types, spellbooks, town service growth loops, class mastery, commendations, contracts, run archives, and durable town unlocks
- Key screenshots: `screenshots/before/mobile-01-creation.png`, `screenshots/before/mobile-03-bank-progression.png`, `screenshots/before/mobile-11-magic-hub.png`, `screenshots/before/mobile-12-level-up-perk.png`, `screenshots/before/mobile-14-relic-choice.png`, `screenshots/after/mobile-progression-atlas-board.png`, `screenshots/after/progression-system-map-board.png`, `screenshots/after/build-path-board.png`
- What this step proved: The current game already has a deep multi-layer progression stack, and build assembly is genuinely one of its strongest pillars. It also proved that the progression story is distributed across many surfaces, which means later redesign work should begin by clarifying emphasis and legibility rather than by adding new systems immediately.
- What the next step should investigate: Determine which current layers are doing the most real work for player identity and long-term motivation, and which ones are currently underwired, under-surfaced, or redundant. Then evaluate whether the progression presentation should be consolidated around clearer run-local versus persistent layers before any mechanical redesign is proposed.

## Step 02: Current Progression Audit

- Folder path: `artifacts/progression-series/step-02-current-progression-audit/`
- Three-sentence summary: This step audits the live progression stack for clarity, payoff, pacing, and dead-choice risk using code inspection plus staged Playwright captures. It ranks the current progression problems by severity, identifies the layers that are already strong enough not to overdesign, and ships three narrow fixes instead of jumping straight to a wholesale redesign. The result is a publication-ready audit trail with before-and-after evidence for the most obvious progression trust repairs.
- Key progression systems documented: level-up flow, random stat growth, spell study, spellbooks, perk identity, relic identity, boon rewards, rumor rewards, objective reward forecasting, extract-or-greed pacing, bank-side persistent progression, and underwired progression entries
- Key screenshots: `screenshots/before/mobile-01-journal-reward-preview.png`, `screenshots/after/mobile-01-journal-reward-preview.png`, `screenshots/before/mobile-02-quick-hands-perk.png`, `screenshots/after/mobile-02-quick-hands-perk.png`, `screenshots/before/mobile-03-greedy-purse-relic.png`, `screenshots/after/mobile-03-greedy-purse-relic.png`
- What this step proved: The game does not currently need more progression layers nearly as much as it needs stronger clarity and a cleanup of underwired choices. It also proved that a few narrow fixes can materially improve trust in the existing system before any larger redesign begins.
- What the next step should investigate: Improve level-up readability around random stat growth, then audit the remaining soft or underwired progression entries such as `Warding Lens`, rumor-oriented rewards, and any currencies or reward types that matter more in code than in felt excitement. After that, decide whether the next redesign step should target reward presentation, bank-side persistence overload, or broader class-path differentiation.

## Step 03: Spell System Deep Dive

- Folder path: `artifacts/progression-series/step-03-spell-system-deep-dive/`
- Three-sentence summary: This step isolates the spell system and evaluates it as its own progression layer, covering taxonomy, acquisition, scaling, tactical identity, and overlap. It shows that the spellbook is already structurally strong, but that too much of its real meaning was buried in code rather than surfaced to the player. The step then ships three focused improvements: richer mechanical spell readouts, stronger progression timing and learn messaging, and a tactical differentiation pass for `Light`.
- Key progression systems documented: spell taxonomy, Spell Study, spellbook acquisition, magic tray usage, spell scaling hooks, utility versus offense patterns, and spell-related build expression
- Key screenshots: `screenshots/before/mobile-01-spell-study.png`, `screenshots/after/mobile-01-spell-study.png`, `screenshots/before/mobile-02-magic-book.png`, `screenshots/after/mobile-02-magic-book.png`, `screenshots/before/mobile-03-field-tray.png`, `screenshots/after/mobile-03-field-tray.png`
- What this step proved: The spell system already supports distinct playstyles through offense, control, sight, travel, defense, and sustain, so the next gains come from clarity and sharper tactical roles rather than from simply adding more spells. It also proved that utility spells become much more valuable once the game explains exact payoff and gives them one clear local job.
- What the next step should investigate: Audit the remaining overlap among `Detect Traps`, `Clairvoyance`, defensive wards, and travel spells, then decide whether late-run spell progression needs better scaling, stronger combo hooks, or more explicit class-path specialization.
