# Teaching Notes

## Why this audit method worked

- Start with the owning modules, not the UI. Progression bugs often look like presentation problems until you confirm whether the data entry is actually wired.
- Separate dead choices from weak copy. Dead choices need code changes; weak copy needs clarity work.
- Capture the same states before and after. That keeps an audit artifact honest and avoids narrating improvements that players cannot actually see.

## Reusable lessons

- If a reward draft depends on one sentence of copy, that sentence has to carry mechanical weight.
- Placeholder reward previews are not harmless. They quietly flatten build planning even when the underlying systems are solid.
- Persistent progression can be rich and still overloaded. Richness and clarity are different axes.
- Narrow fixes are valuable in an audit phase when they restore trust in the existing system without pre-committing to a larger redesign.
