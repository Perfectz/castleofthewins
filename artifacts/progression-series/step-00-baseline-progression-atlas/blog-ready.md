# Baseline Progression Atlas

Before changing a progression system, it is worth proving what is already there. This baseline pass does that for the current browser build of *Castle of the Winds*: not as a redesign pitch, but as a code-backed atlas of how abilities, spells, perks, relics, boons, unlocks, and next-run systems currently work.

The headline finding is that the game already has a richer progression stack than a quick play session suggests. Character growth is not just XP and items. Race and class define the opening shape. Level-ups trigger both random stat growth and a two-part choice sequence: first a class-family perk, then a spell study choice. Spellbooks offer a second acquisition route for spells outside level-up. Objective rewards branch into relic, boon, or rumor outcomes. Then the bank layer carries value forward through mastery, commendations, contracts, town unlocks, and run archives.

That layered structure is a strength. It means the game already supports build assembly through several parallel systems instead of through one dominant upgrade ladder. A Fighter can lean harder into guard and survival. A Rogue can steer toward scouting, control, and pressure discipline. A Wizard can compound spell quality, mana economy, and overcast tolerance. Gear, spellbooks, town access, and floor rewards all push those paths around in useful ways.

The baseline also found an important structural truth: progression is distributed across many surfaces. The creation screen owns identity. The armory and guild own acquisition. The pack owns gear fit and spellbook use. The magic hub owns spell management. Level-up modals own perk and spell study decisions. Objective rewards own relic and boon branches. The bank owns persistence review, mastery, contracts, commendations, rumors, and town projects. That distribution creates depth, but it also means the full progression story is easy to miss unless the player reads several different surfaces carefully.

Playwright helped make that visible. The screenshot pass captured character creation, town progression services, inventory build shaping, the magic hub, level-up perk choice, spell study, relic choice, and boon choice in both phone-sized and desktop layouts. Those raw captures are useful evidence, but the after-side boards are what make them readable. The phone and desktop progression atlas boards compress the baseline into one scanable artifact. The system map board clarifies the relationship between run-local and persistent growth. The build-path board summarizes the current Fighter, Rogue, and Wizard growth shapes without pretending a redesign has already been chosen.

The code pass surfaced a second important truth: some progression content is more fully defined than fully realized. A few perks, relics, or currencies appear clearly in data but are not as obviously wired into the gameplay layer. That is not a criticism for its own sake. It is exactly the kind of baseline fact that should influence redesign scope later. If a future redesign invests heavily in a system that is still only lightly implemented, it risks polishing a promise instead of strengthening a proven pillar.

So what did this step prove? First, the current game already has enough progression depth to justify a serious redesign study instead of a superficial UI pass. Second, the strongest existing pillar really is build assembly, because multiple independent systems are already feeding it. Third, the next redesign step should not begin by inventing new progression layers. It should begin by deciding which current layers are carrying the clearest player value, which ones are under-surfaced, and which ones are only half-activated today.

That is what a baseline is for. It keeps the next step honest.
