# Teaching Notes

Establishing a baseline before redesign is an advanced prompting best practice because it keeps the work grounded in the shipped product. Without a baseline, redesign prompts tend to optimize against guesses, isolated complaints, or the model's default taste instead of the actual experience players see.

Why this matters:

- A baseline separates observation from invention. First you document what exists, then you decide what should change.
- A baseline exposes the difference between reachable states and staged states. That distinction matters when the redesign later claims to simplify real player flows.
- A baseline creates reusable evidence. The same screenshots, boards, and manifest can support design critique, implementation planning, QA, and public communication.

In prompting terms, this step improves later work because it gives future redesign prompts a concrete corpus:

- named surfaces
- stable screenshot references
- responsive comparisons
- documented weak points
- clear notes about what was and was not reachable

That is a higher-signal starting point than "make the UI cleaner" or "redesign the menus." It constrains the next prompt with real evidence while still leaving space for strong creative direction.
